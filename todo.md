# Todo

## Plan
Implement the core Epoxy library top-down, starting with the most important user-facing feature: the model+controller+adapter system with automatic diffing. Then layer on annotations, model groups, typed controllers, touch support, and visibility tracking. Each task delivers independently testable functionality.

## Tasks
- [ ] Task 1: Implement the core item model and controller with automatic list diffing — Create a model class representing a list item with unique ID, properties, equality/hashing, show/hide, immutability enforcement, and bind/unbind lifecycle. Create a specialized change-tracking list with freeze support. Create a diff algorithm that computes minimal insert/remove/move/update operations between model lists, with batched update operations. Create a controller where users override buildModels() to declare items, with request-based rebuilding, interceptors, duplicate filtering, and model build listeners.
- [ ] Task 2: Implement the list adapter with view holder management and view type mapping — Create an adapter that manages view holders, maps view types to model classes, saves/restores view state, and dispatches model lifecycle events. Include both a direct-manipulation adapter (where users modify models list directly and call notify methods) and a controller-backed adapter that integrates with the diff algorithm for automatic updates.
- [ ] Task 3: Implement annotation definitions for model attributes, views, and package configuration — Create annotation types for marking model attributes with code generation options (hash, getter, setter control), view-based model generation with layout and sizing options, text and callback property shortcuts, lifecycle hooks (after props set, on recycle, visibility change), auto-model fields in controllers, data binding layout declarations, and package-level configuration for validation and generation options.
- [ ] Task 4: Implement model groups for compositing multiple child models into a single list item — Allow users to group several models together as a single item, with proper delegation of bind/unbind lifecycle to child models, child view recycling via a shared pool, and support for different layout strategies (direct child views or view stubs).
- [ ] Task 5: Implement typed controllers for data-driven model building with 1-4 type parameters — Create controller variants that accept typed data via setData() methods, automatically triggering model rebuilds when data changes. Provide variants for 1, 2, 3, and 4 data types, plus a simple controller that accepts pre-built model lists, and an async controller that runs building and diffing on background threads.
- [ ] Task 6: Implement touch interaction support for drag-and-drop reordering and swipe-to-dismiss — Create a builder-style API for configuring which model types support dragging and swiping, with customizable callbacks for drag start/release, swipe completion, movement direction constraints, and clear/reset actions.
- [ ] Task 7: Implement visibility tracking to monitor item visibility state changes during scrolling — Track visibility of items with state transitions (visible, invisible, focused, unfocused, partial impression, full impression) and percentage-based thresholds, supporting both flat lists and nested lists with proper deduplication of state notifications.
